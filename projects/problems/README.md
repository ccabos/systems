# Problems

Problem-driven application area. Each subdirectory is one concrete
problem that the SE-for-non-technical-systems toolkit is being applied
to. This is where the knowledge base (`../../knowledge/`) meets a
specific situation with a specific stakeholder and a specific
question.

## Folder convention

Every problem folder must contain a `problem.md` at its root that
describes the problem clearly — the stakeholder, the goal, the
observed symptoms, and the constraints. Everything else in the folder
is derivative: analyses, drafts, diagrams, interview notes, partial
decompositions, remedy proposals.

The rule of thumb from `../README.md` applies: problems may reference
and excerpt from `knowledge/`, but must not contain new canonical
decompositions. If a problem requires decomposing a new system or
defining a new technique, that work happens in `knowledge/` first and
is then cited from here.

## Starting a new problem

Copy `_template/` to a new folder named after the problem (short,
kebab-case, no stakeholder PII). Fill in `problem.md` first; the rest
of the template files are optional and should only be created if the
problem actually needs them.
